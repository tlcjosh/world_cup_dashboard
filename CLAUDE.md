# World Cup 2026 Dashboard — Claude Context

## What This Is
A "Dynamic Static" single-page app tracking FIFA World Cup 2026. The frontend polls ESPN's public scoreboard API every 10 seconds directly from the browser for live scores. GitHub Actions backs this up by polling ESPN every 5 minutes and committing `src/data/data.json`, which the frontend uses for the full match schedule and standings. football-data.org is schedule scaffolding only (bootstrap + `matchId` self-heal) — see "Score Source of Truth" below for why it was removed from the live-score path. No backend server.

### Match Card Layout
`.match-card` is a flex column with this structure:

1. **`.match-meta-bar`** — `space-between` flex row:

   - Right: `.match-meta-right` — venue text (small, muted)

2. **`.match-teams`** — 3-column CSS grid: `1fr auto 1fr`
   - `.match-home` — flex row, right-aligned: team name then flag
   - `.score-col` — centered; contains `.score` (30px Anybody) and optional `.score-sub.live.match-clock`
   - `.match-away` — flex row, left-aligned: flag then team name

3. **`.match-events`** (optional) — 2-column grid for goal scorers: `.me-home` left, `.me-away` right. Each scorer is a `.goal-event` span with ⚽ icon. Rendered by `espnEventsHtml()` when match has scored events.

4. **`.match-stats`** (optional) — inset pill (`rgba(0,0,0,.03)` bg, border-radius 8px) containing:
   - `.stats-poss` grid: home %, possession bar (two-tone team-color gradient), away %, "Possession" label
   - `.stats-grid`: shot/corner/card rows via `.sg-h` / `.sg-l` / `.sg-a`
   - `.ycard` / `.rcard` — yellow/red card icons with counts
   Rendered by `espnStatsHtml()` for any match that has ESPN stat data (not just live).

5. **`.match-headline`** (optional) — italic ESPN recap summary sentence.

6. **Live commentary** (optional, in-play only) — most recent comment from ESPN summary endpoint, rendered directly on the card's white background beneath the stats block (no fill color or border), with a 📰 icon (`.mc-icon`) to the left of the text and `‹`/`›` scroll buttons (`.mc-btn`) plus an `N/total` counter (`.mc-count`) to step back through older comments. Fetched separately via `fetchESPNCommentary()` for each in-play match that has an `espnEventId`; the full commentary history is merged by `sequence` into `match._espnCommentary` (no cap). Rendered by `commentaryInnerHtml()`, which tracks the displayed comment by `match._commentarySeq` (not a raw index) so new comments arriving mid-scroll never yank the view away from what's being read. After 15s of no manual scroll, `scheduleCommentaryResume()` snaps the view back to the latest comment. The fade-in on text change is the `mc-fade-in` keyframe animation, replayed via the `.mc-anim` class toggle.

Live cards get a CSS gradient border via `background: linear-gradient(surface, surface) padding-box, grad-live border-box`.

Winner/loser classes on team names: `.team-name.winner` (bold) / `.team-name.loser` (muted).

### Dashboard Hero Stats
All three hero slots rotate together through a pool built by `computeHeroStats()`: Played, Goals scored, Goals/match, Clean sheets, Highest-scoring match, Cards shown, Fouls committed, Saves, Corners won, Pass accuracy, Shots on target, Tackles won, Interceptions. "Played" and the always-available goals/clean-sheets entries are unconditional; the rest are only added to the pool once at least one match has that data (`cardsKnown`/`foulsKnown`/`savesKnown`/`cornersKnown`/`passPctKnown`/`shotsOnTargetKnown`/`tacklesKnown`/`interceptionsKnown` counters) — early in the tournament, before any ESPN stats exist, the pool falls back to just the always-available entries.

- **Rotation**: `state._heroStatIdx` is a module-level counter (persisted across renders, not reset per poll), advanced by 3 (not 1) on each tick of its own `setInterval` (24s, set up in `init()` as `state.heroStatInterval`) that calls `renderView({ silent: true })`. Advancing by the full window size means all three displayed stats are replaced together each tick rather than sliding one at a time. Since rendering goes through `morphInto`, only the changed `.hero-stat-num`/`.hero-stat-label` text nodes patch — no flicker, same idiomorph diffing used elsewhere. Three pool entries are shown at once: `pool[idx % pool.length]`, `pool[(idx+1) % pool.length]`, `pool[(idx+2) % pool.length]`. This interval doesn't need restarting on PWA resume (unlike the ESPN/data.json poll intervals) since it's purely cosmetic — a few missed rotations while backgrounded is harmless.
- **Data source**: goals/clean-sheets/highest-scoring-match come straight from `homeScore`/`awayScore`, always present on any `FINISHED` or in-progress match. Cards/fouls/saves come from the `homeYellowCards`/`homeFouls`/`homeSaves`-style fields described in "In-memory match fields added by ESPN" above — live preview while ESPN tracks a match, permanent once `update_tracker.js`'s `syncMatchStats()` bakes them in.

## Live Score Architecture

### Primary: ESPN API (browser-side, every 10s)
`fetchESPN()` in `app.js` polls:
```
https://site.api.espn.com/apis/site/v2/sports/soccer/fifa.world/scoreboard
```
No API key required. CORS-friendly — works directly from browser JS. Returns only **today's** matches, so it overlays live data on top of the full schedule from `data.json`.

`mergeESPNData()` matches ESPN events to `state.matches` by **ESPN team ID first** (`ESPN_ID_TO_TEAM` reverse map from `TEAM_MASTER_DATA[team].espnId`), falling back to display name normalization via `ESPN_NAME_MAP`. This ID-first approach avoids name mismatch issues. After matching, enriches each match with:
- `status`, `homeScore`, `awayScore` — always updated
- `espnEventId` — ESPN's match ID, used for commentary fetches and historical stat lookups
- `_espnClock`, `_espnDisplayClock`, `_espnPeriod`, `_espnFetchedAt` — live clock: elapsed seconds, formatted string (e.g. `"45'+2'"`), period, fetch timestamp
- `_espnEvents` — goal scorer labels parsed from `competitions[0].details[]` (scoring plays only), e.g. `"E. Ashour 20'"`, `"M. Hany 66' (og)"`
- `_espnStats` — every key from ESPN's `competitor.statistics[]` array, copied through generically (not an allowlist) plus card counts derived from `details[]`. Includes `possessionPct`, `totalShots`, `shotsOnTarget`, `wonCorners`, `yellowCards`, `redCards`, and others not currently surfaced anywhere (`offsides`, `penaltyKickGoals`, `totalTackles`, `blockedShots`, `interceptions`, etc. — available if a future stat needs them). **Does not include `saves` or `passPct`** — those two fields don't exist anywhere in the scoreboard endpoint's stats array (verified against `blueprint_data`); they only live in the `/summary` endpoint's `boxscore.teams[]`, sourced separately — see `homeSaves`/`homePassPct` below and "ESPN Live Commentary".
- `_espnColors` — hex brand colors for each team; auto-falls back to `alternateColor` when primary is near-white. Used for the possession gradient bar.
- `_espnHeadline` — recap text from `competitions[0].headlines[0].description`
- `homeFairPlay`, `awayFairPlay` — FIFA fair-play disciplinary deduction for this match only (e.g. `-1`, `-4`), derived from `competitions[0].details[]` card events via `classifyMatchFairPlay()`. Live preview while ESPN is tracking the match; see "Fair Play Tiebreaker" below.
- `homeYellowCards`, `awayYellowCards`, `homeRedCards`, `awayRedCards`, `homeFouls`, `awayFouls`, `homeCorners`, `awayCorners`, `homeShots`, `awayShots`, `homeShotsOnTarget`, `awayShotsOnTarget`, `homeTackles`, `awayTackles`, `homeInterceptions`, `awayInterceptions`, `homeClearances`, `awayClearances`, `homeCrosses`, `awayCrosses`, `homeLongBalls`, `awayLongBalls` — raw per-match counts promoted to top-level fields from `_espnStats`/the card-counting loop above, every 10s scoreboard poll. Live preview while ESPN tracks the match; permanent once `update_tracker.js`'s `syncMatchStats()` bakes them in after `FINISHED`.
- `homeSaves`, `awaySaves`, `homePassPct`, `awayPassPct` — same promoted-field pattern, but sourced from `boxscore.teams[]` in the `/summary` endpoint instead (see `_espnStats` note above), set by `fetchESPNCommentary()` rather than `mergeESPNData()`'s main loop. Only refreshed for in-play/paused matches (since that's the only time `fetchESPNCommentary()` runs); permanent once `syncMatchStats()` bakes them in after `FINISHED`. All six fields feed the Dashboard hero's rotating stat pool — see "Dashboard Hero Stats" below.

**Swapped teams**: When ESPN's home/away order differs from `data.json`, a `swapped` flag reverses all score/stat/color assignments.

**Important**: `mergeESPNData()` always calls `renderView()` after every sync (not just when scores change), so stats and headlines appear immediately on first page load even when all matches are already FINISHED.

### ESPN Live Commentary (per in-play match)
After each scoreboard poll, `fetchESPNCommentary(match)` is called for every in-play match with an `espnEventId`. Hits the ESPN summary endpoint:
```
{ESPN_SCOREBOARD_URL.replace('/scoreboard', '/summary')}?event={espnEventId}
```
Parses `data.commentary[]`, merges into `match._espnCommentary` by `sequence` (a `Map` keyed by sequence, so re-fetches dedupe instead of appending duplicates), sorted most-recent-first. No cap — the full commentary history for the match is retained, since ESPN's summary endpoint returns the whole feed on every poll anyway, not just deltas. Renders on live match cards directly on the white card background (no special background/border) beneath the stats block, with a 📰 icon to the left of the text. Fails silently if unavailable.

Since this is already a per-match fetch of the `/summary` endpoint, it also piggybacks `data.boxscore.teams[]` to set `homeSaves`/`awaySaves`/`homePassPct`/`awayPassPct` — see the note under "Primary: ESPN API" above for why those two stats need this endpoint instead of the scoreboard one. Matched to home/away by `TEAM_MASTER_DATA[team].espnId` against `boxscore.teams[].team.id`, same ID-first pattern used everywhere else.

**Scrolling through commentary**: each card shows one comment at a time (latest by default) plus `‹`/`›` buttons to step through the full buffered history. A delegated click handler (`.mc-btn`) finds the match by `data-matchnum`, moves `match._commentarySeq` to the next/previous item's `sequence`, re-renders just that card's `.match-commentary` node (via `commentaryInnerHtml()`) rather than the whole view, replaying the `mc-fade-in` animation, and calls `scheduleCommentaryResume(matchNum)` (unless the user just navigated back to the latest comment, in which case any pending resume timer is cleared instead). `scheduleCommentaryResume()` sets a 15s (`COMMENTARY_RESUME_MS`) timeout — tracked per-match in the module-level `commentaryResumeTimers` Map, re-looking up the match by `matchNum` when it fires rather than closing over a possibly-stale match object — that snaps the card back to the latest comment if the user hasn't navigated again within that window. `commentaryInnerHtml()` also re-checks the elapsed time on every regular render (not just the timer), so the snap-back still happens even if the `setTimeout` gets dropped (e.g. backgrounded-tab throttling).

**Important**: `_commentarySeq` is only ever set to a concrete `sequence` value while scrolled away from the latest comment (idx > 0). Whenever the view lands back on the latest comment — via the `›` button, the 15s auto-resume, or the timestamp fallback — `_commentarySeq` is reset to `null`/`undefined` rather than pinned to that comment's `sequence`. This matters because `_espnCommentary` keeps growing as new commentary arrives: pinning to a snapshot `sequence` would freeze the view on "today's latest" forever once one more comment came in, since nothing re-arms the resume timer once you're already considered "caught up." Leaving `_commentarySeq` unset means `commentaryInnerHtml()`'s default (`idx === -1 → idx = 0`) keeps tracking whatever is actually newest, indefinitely, until the user explicitly scrolls away again. `_commentarySeq` (when set) is preserved across `mergeESPNData()` (object mutated in place) and across `fetchData()`'s `data.json` merges (listed in `ESPN_FIELDS`), so a user's scroll position survives both the 10s ESPN poll and the 2-minute data.json refresh.

### Penalty Shootouts
Confirmed against a real live shootout (Netherlands v Morocco, R32, 2026-06-29) using a one-off GitHub Actions debug harness (`scripts/debug_espn_dump.js` + `.github/workflows/debug-espn.yml`, kept in the repo as a `workflow_dispatch`-only tool for dumping raw ESPN JSON on demand — not part of the regular pipeline).

ESPN's status flow for a match that finishes level after extra time: `STATUS_END_OF_EXTRATIME` (period 4, detail "AET-pens") for the brief gap before kicks start, then `STATUS_SHOOTOUT` (period 5, detail "Penalties"/"Pens") for the shootout itself. `displayClock`/`clock` freeze at `"120'"`/`7200` throughout — there's no running clock to show, so `getMatchMinute()` returns `null` for period 5 and the score-sub area shows the shootout score instead (see below). Both statuses map to `IN_PLAY` in `ESPN_STATUS_MAP`; the eventual `STATUS_FINAL_PEN` (already mapped to `FINISHED`) ends it.

- **Source**: the scoreboard/`details[]` endpoint flags shootout kicks with a boolean `shootout: true` (vs. `false` for every normal-time/extra-time goal/card event) and `penaltyKick: true`, `scoreValue: 1` per made kick — `mergeESPNData()`'s goal-scorer loop explicitly skips `d.shootout` entries so shootout kicks never get mis-rendered as match goals or fire a goal notification. The actual live shootout data comes from a **different, dedicated source**: the `/summary?event=` endpoint's top-level `shootout[]` array, present only once the shootout begins (absent during the period-4 AET-pens gap):
  ```json
  "shootout": [
    { "id": "449", "team": "Netherlands", "shots": [
      { "playerId": "258968", "player": "Teun Koopmeiners", "shotNumber": 1, "didScore": true },
      { "playerId": "245916", "player": "Justin Kluivert", "shotNumber": 2, "didScore": false }
    ] },
    { "id": "2869", "team": "Morocco", "shots": [ /* same shape */ ] }
  ]
  ```
  Confirmed stable and incrementally growing (one new `shots[]` entry per kick) across repeated live polls. `id` is ESPN's numeric team id (matches `TEAM_MASTER_DATA[team].espnId`), same ID-first matching pattern used everywhere else.
- **Pitfall confirmed by direct observation, including against two real completed shootouts**: `boxscore.teams[].statistics[]`'s `penaltyKickGoals`/`penaltyKickShots` do **NOT** update during an in-progress shootout (stayed `"0"`/`"0"` after multiple made kicks on both sides) — and, contrary to the original assumption, **never finalize even once the match is fully `FINISHED`**. Re-checked directly against Germany v Paraguay and Netherlands v Morocco (both 2026-06-29, both `STATUS_FINAL_PEN`/`completed: true`) via the `debug-espn.yml` harness: both teams still show `penaltyKickGoals: "0"` in the boxscore for both matches. This stat appears to simply never populate for a shootout-decided match. `shootout[]` is therefore the **only** working source, live or permanent — there is no live/permanent split for this field like there is for saves/passPct.
- **Frontend (live preview)**: `fetchESPNCommentary()` in `app.js` — already doing a per-match `/summary?event=` fetch for commentary/saves/passPct — also reads `data.shootout[]`, matched to home/away by `espnId`, and sets `match._shootout = {home: shots[], away: shots[]}` plus `homeShootoutScore`/`awayShootoutScore` (counts of `didScore === true`). Both are listed in `ESPN_FIELDS`/`ESPN_AUTHORITATIVE_FIELDS` so they survive the 2-minute `data.json` merge.
- **Backend (permanent)**: `update_tracker.js`'s `syncMatchStats()` bakes in `homeShootoutScore`/`awayShootoutScore` the same way the frontend does — counting `didScore === true` entries in the `/summary` endpoint's `shootout[]` array (`parseShootoutScore()`), matched by ESPN team id — **not** from any boxscore stat, per the pitfall above. `isStatsComplete()`'s completeness gate additionally requires `homeShootoutScore` to be a number whenever `wentToShootout(m)` is true (knockout stage + `homeScore === awayScore`, since a knockout match can only finish level via penalties), so a match isn't marked "fully synced" until its shootout score is actually backfilled.
- **Display**: `statusBadge()` shows a `PENS` badge for period 5 (alongside the existing `ET` for periods 3/4). The score-sub area shows `PENS h–a` instead of a clock. `espnShootoutHtml()` renders a compact kick-by-kick row (`.match-shootout`) below the goal-scorer line — green/red filled/hollow circles per kick, named via a `title` tooltip — whenever `match._shootout` has data, live or already finished.
- **Bracket resolution — the actual reason this matters**: a knockout match decided on penalties stays tied `homeScore === awayScore` forever (the AET scoreline), so the `[W##]`/`[L##]` placeholder resolution in both `resolveTeam()` (`app.js`, frontend speculative/live) and `resolveBracketPlaceholders()` (`update_tracker.js`, backend permanent) now checks `homeShootoutScore`/`awayShootoutScore` as a tiebreaker whenever the regular score is tied, before falling back to "unresolved." **This was a real, previously-shipped bug**: the backend's `resolveBracketPlaceholders()` used to compute `homeWon = feeder.homeScore > feeder.awayScore` unconditionally, which is always `false` for a shootout-decided tie — i.e. it would have silently advanced the *loser* of every penalty shootout into the next round. If any earlier R32 match already went to penalties and got its bracket slot resolved before this fix shipped, double-check that match's downstream `[W##]`/`[L##]` placeholders resolved to the correct (winning) team — `resolveBracketPlaceholders()` only re-derives a slot while it's still a placeholder string, so a wrongly-resolved name won't self-correct on a later sync.

### Extra-Time Transition Gap
Confirmed live 2026-07-01, Belgium v Senegal: the dashboard visibly stalled at `90'+8'` for several minutes between the end of regulation and the start of extra time (ESPN's own clock genuinely freezes here — there's no running clock during this real-world break, same as the AET-pens gap in "Penalty Shootouts" above), then later briefly showed a plain `HT` badge for the short break between the two extra-time halves. Both are the same underlying gap in fidelity: `PAUSED` collapses several visually distinct real-world moments (real half-time, the post-90'-decision gap, the between-ET-halves break) into one status, and the badge/clock code used to hardcode `HT` for all of them regardless of `_espnPeriod`.

- **`pausedStatusLabel(match)`** (`app.js`) is the single source of truth for the label shown whenever `match.status === 'PAUSED'`, used by `statusBadge()`, `getMatchMinute()`, and `matchCardHtml()`'s score-sub so all three agree:
  - `_espnPeriod` 3 or 4 → `"ET BREAK"` (between extra-time halves, or the brief AET-pens gap if it ever surfaces as `PAUSED` instead of `STATUS_END_OF_EXTRATIME`'s `IN_PLAY` mapping)
  - `_espnPeriod` 2 **and** `match._regulationEnded` → `"BREAK"` (regulation just ended, ET/pens decision pending)
  - otherwise → `"HT"` (real half-time, or an ordinary mid/end-of-second-half stoppage that hasn't been confirmed as the post-regulation gap)
- **`_regulationEnded`** is set by `fetchESPNCommentary()` from `data.keyEvents[]` — a sibling of `commentary[]` in the same `/summary?event=` response already being fetched for every in-play/paused match, previously never read. Its `"end-regular-time"` entry (`type.type === 'end-regular-time'`, confirmed against a captured group-stage match in `blueprint_data/espn_example_endpoints.json`) fires exactly once, the instant the ref blows for the end of the 90 minutes, independent of whatever the status field is doing — this is the same event that produces the human-readable `"Second Half ends, ‹home› ‹score›, ‹away› ‹score›."` commentary line. Listed in `ESPN_FIELDS` so it survives the `data.json` merge in `fetchData()`, same as `_espnPeriod` etc.
- This intentionally does **not** try to guess the exact ESPN status name active during the period-2 gap (unconfirmed — see `scripts/snapshot_extra_time.js` below for how we'll catch it next time) or distinguish it from an ordinary mid-second-half stoppage (e.g. the weather delay in Known Issue #13) by any means other than the `end-regular-time` marker — a plain stoppage before 90' never fires that event, so `_regulationEnded` stays falsy and the badge correctly falls back to `"HT"`.
- Since this only fires once per match and only matters while `PAUSED`, no reset/decay logic is needed — once `_espnPeriod` advances past 2, `pausedStatusLabel()` stops consulting the flag at all.

### Extra-Time Stoppage Clock Freeze
Confirmed live 2026-07-01, Belgium v Senegal (second half of extra time): the dashboard hung on `"120'"` for the entire ~12-minute stoppage period instead of showing added time. Root cause, proven directly from captured data already sitting in `blueprint_data/espn_scoreboard_specific_date.json`/`espn_api_example.json`: ESPN's raw `status.clock` (seconds) field **caps at the period boundary and never advances past it**, even once real stoppage minutes are played — e.g. several `STATUS_FULL_TIME` matches in that data show `clock: 5400` (90:00) alongside a correctly-incrementing `displayClock: "90'+9'"`. For periods 1/2, `getMatchMinute()` already prefers `displayClock`'s `"+N"` suffix during stoppage, so this is invisible there. Periods 3/4 (extra time) have no equivalent — the 2026-06-29 fix for the "stuck at 90+X during ET" bug (commit `08a5083`) deliberately disabled the `displayClock` fallback for ET, since it was observed to keep anchoring to `"90'+N'"` instead of relabeling to 105'/120', and fell back to raw-clock-only. That raw clock inherits the exact same capping behavior with nothing to catch it.

- **Fix**: `mergeESPNData()` now tracks `_clockFrozenSince` — the wall-clock moment the raw `_espnClock` value (and `_espnPeriod`) last actually changed between polls. `getMatchMinute()`'s period 3/4 branch, once `_espnClock` is at or past the period's cap (`base * 60` seconds) **and** `_clockFrozenSince` is set, counts real elapsed time from that timestamp (`Math.floor((Date.now() - _clockFrozenSince) / 60000)`) instead of freezing at `"105'"`/`"120'"` forever — same idea as the `_regulationEnded` fix above: don't wait on an ESPN field known to go stale exactly when it matters most.
- **False-positive guard**: the frozen-clock branch only engages once `_espnClock >= base * 60` — a one-off feed hiccup earlier in the half (two polls coincidentally reporting the same clock value well before stoppage) just falls through to the normal raw-elapsed-time calculation and self-corrects on the next poll, rather than prematurely showing `"105+N'"`/`"120+N'"`.
- **This is an estimate, not ESPN's official added-time figure** — we don't know how many minutes the referee actually announced during ET stoppage (unlike periods 1/2, where `displayClock` gives the real number). It guarantees the clock visibly keeps moving instead of looking dead, at the cost of not necessarily matching the broadcast's official countdown to the minute. Listed in `ESPN_FIELDS` so it survives the `data.json` merge in `fetchData()`, same as `_espnClock` etc.
- **Not caused by the page reload** the user suspected when this was first reported — a reload just re-fetches the same already-frozen upstream ESPN value on the very next poll, so it looks like reloading "doesn't help," which is consistent with (not contradictory to) this being an upstream data gap rather than a client-side state bug.
- Deliberately does **not** depend on `scripts/snapshot_extra_time.js` ever capturing a live ET payload to confirm exactly what `displayClock` contains during ET stoppage — that harness was considered but assessed as not load-bearing for this fix (the wall-clock fallback works regardless of what `displayClock` says), so it wasn't built out further. See git history around 2026-07-02 if that assessment ever needs revisiting.

### ESPN Date Cache (historical data)
`fetchESPNDate(dateStr)` fetches `?dates=YYYYMMDD` and caches the result in `state._espnDateCache[dateStr]`. Used by the **match detail modal** (not the team profile modal, see "Team Profile Modal" above) to get the full goal/card timeline-with-minutes for a single historical match — that level of play-by-play detail isn't persisted to `data.json`, only per-match aggregate stats are, so this is the one remaining place a finished match still triggers a live ESPN fetch.

### Fallback: data.json (every 2 minutes)
`fetchData()` re-fetches `data.json` every 2 minutes to pick up schedule changes, knockout match updates, and standings corrections.

- **Source-of-truth precedence**: while ESPN is reachable (`state.espnSynced`), it's authoritative for any match it's actively tracking (`espnEventId` set) — `status`/`homeScore`/`awayScore` plus clock/stats/events/commentary fields (`ESPN_FIELDS` / `ESPN_AUTHORITATIVE_FIELDS` in `fetchData()`) are preserved from the in-memory match rather than overwritten by data.json. data.json becomes authoritative only once ESPN is unreachable, or for matches ESPN's date-scoped scoreboard doesn't cover at all.
- **No-op skip**: if `data.lastUpdated` is unchanged since the last fetch (most 2-minute ticks, since Actions only commits every ~5 minutes), `fetchData()` returns immediately without rebuilding matches/standings or re-rendering.
- `combinations.json` is fetched once at startup (`fetchCombinations()`), not on every poll — it's static and never changes at runtime.

### PWA Resume Handling
Installed Android PWAs throttle or fully suspend `setInterval` timers while backgrounded (screen off, app switched away), leaving the UI on stale data until the next throttled tick. `resyncNow()` forces an immediate `fetchData()` + `fetchESPN()` and restarts both poll intervals (so their next tick is measured from "now"), triggered by:
- `visibilitychange` → `visible`
- `online` event
- `pageshow` with `event.persisted` true (back/forward-cache restore — the common case when resuming an installed PWA)

### ESPN Team Name Map
`ESPN_NAME_MAP` in `app.js` normalizes ESPN display names to our `TEAM_MASTER_DATA` keys. Known mismatches:
- `"Cape Verde"` → `"Cape Verde Islands"`

Add new entries here as mismatches are discovered during the tournament. ID-based matching (`espnId` in `TEAM_MASTER_DATA`) takes precedence and bypasses this map entirely.

### ESPN Status Map
ESPN status names → our internal status values:
- `STATUS_SCHEDULED` → `SCHEDULED`
- `STATUS_FIRST_HALF` / `STATUS_SECOND_HALF` → `IN_PLAY`
- `STATUS_HALFTIME` / `STATUS_END_PERIOD` / `STATUS_SUSPENDED` / `STATUS_DELAY` / `STATUS_DELAYED` → `PAUSED`
- `STATUS_END_OF_EXTRATIME` (period 4, brief AET-pens gap) / `STATUS_SHOOTOUT` (period 5, shootout in progress) → `IN_PLAY` — see "Penalty Shootouts" above
- `STATUS_FULL_TIME` / `STATUS_FINAL_AET` / `STATUS_FINAL_PEN` → `FINISHED`
- `STATUS_POSTPONED` / `STATUS_CANCELED` → `SCHEDULED`

## Data Structures

### `data.json` match object
```json
{
  "matchId": 537327,
  "matchNum": 1,
  "stage": "Group Stage",
  "group": "A",
  "homeTeam": "Mexico",
  "homeIso": "mx",
  "awayTeam": "South Africa",
  "awayIso": "za",
  "venue": "Estadio Azteca, Mexico City",
  "kickoff": "2026-06-11T19:00:00.000Z",
  "homeScore": 2,
  "awayScore": 0,
  "status": "FINISHED",
  "firstHalfStart": null,
  "secondHalfStart": null
}
```
- All timestamps are UTC ISO strings
- Scores are `null` for unplayed matches, integers (including 0) when played
- Status values: `FINISHED` | `IN_PLAY` | `PAUSED` | `SCHEDULED`
- `group` on a match is set from the home team's group (for display in Schedule). Cross-group matches (e.g. Spain G vs Saudi Arabia H) will show under the home team's group.
- Knockout matches use placeholder strings for teams: `[1A]`, `[2C]`, `[3ABCDF]`, `[W73]`, `[L101]`
- `matchNum` 1–72 = Group Stage; 73–88 = R32; 89–96 = R16; 97–100 = QF; 101–102 = SF; 103 = 3rd Place; 104 = Final

| Field | Type | Description |

### `data.json` standings object
```json
{
  "A": [
    { "team": "Mexico", "iso": "mx", "played": 1, "won": 1, "drawn": 0, "lost": 0, "gf": 2, "ga": 0, "gd": 2, "pts": 3, "fairPlayPoints": -1 }
  ]
}
```
Teams sorted: pts → gd → gf → head-to-head mini-league → fair play → FIFA World Ranking → team name alphabetically. `fairPlayPoints` is the sum of `homeFairPlay`/`awayFairPlay` across the team's matches, computed by `computeStandings()`.

## Tournament Logic

### Group Source of Truth
`computeStandings()` in both `update_tracker.js` and `app.js` uses `TEAM_MASTER_DATA[team].group` to determine each team's group — NOT the match's `group` field (which is only set once, cosmetically, at match creation for the Schedule view's group label). This is defensive: it guarantees standings are always computed from the single source of truth for group membership, regardless of what's cached on a given match record.

ESPN's `altGameNote` field (e.g. `"FIFA World Cup, Group H"`) reflects the home team's group — consistent with our `data.json` convention.

**Note on history**: an earlier version of this doc described a "Cross-Group Matches (WC2026 format)" quirk where Groups G and H supposedly played cross-group fixtures in rounds 2–3. That was a misdiagnosis — there is no such format quirk in WC2026. It was a previous Claude session rationalizing the symptom of a real data bug: `TEAM_MASTER_DATA` had Spain/Cape Verde Islands and Iran/New Zealand swapped between groups G and H (see Known Issues below). Once the swap was corrected, every match in the schedule resolves to a standard intra-group fixture — zero actual cross-group matches exist anywhere in the 104-match schedule.

### Group Assignments (Groups G & H — easy to mix up, see Known Issues)
- **Group G**: New Zealand, Iran, Belgium, Egypt
- **Group H**: Uruguay, Saudi Arabia, Spain, Cape Verde Islands

### 3rd-Place Wildcard
8 of 12 third-place teams advance. Logic in `app.js` → `computeThirdPlaceRankings()`:
1. Get position-3 team from each group's standings
2. Sort all 12 by pts → gd → gf → fair play → FIFA World Ranking (no head-to-head — these teams are in different groups and never play each other in the group stage)
3. Top 8 qualify; their group letters sorted alphabetically = combination string (e.g. `"ABCDEFKL"`)
4. `combinations.json[combString]` maps `"1A"` ... `"1L"` → `"3X"` team codes

### Bracket Slot Mapping
Each 3rd-place bracket slot placeholder determines which combinations.json column to look up:
```
"[3CEFHI]" → faces 1A → combinations["1A"]
"[3EFGIJ]" → faces 1B → combinations["1B"]
"[3BEFIJ]" → faces 1D → combinations["1D"]
"[3ABCDF]" → faces 1E → combinations["1E"]
"[3AEHIJ]" → faces 1G → combinations["1G"]
"[3CDFGH]" → faces 1I → combinations["1I"]
"[3DEIJL]" → faces 1K → combinations["1K"]
"[3EHIJK]" → faces 1L → combinations["1L"]
```
Defined in `app.js` as `SLOT_TO_OPPONENT`.

### Bracket Visual Ordering
`BRACKET_TEMPLATE`'s `matchNum` order (73–104) is chronological by kickoff date, **not** bracket-tree order — R32 match 74 doesn't necessarily feed R16 match 89 just because it's numerically adjacent. `renderBracket()` reorders each round's column for display using two hardcoded tree-order constants verified against the `[W##]` feeder placeholders in `BRACKET_TEMPLATE` and FIFA's official bracket:
```js
const R32_TREE_ORDER = [74, 77, 73, 75, 83, 84, 81, 82, 76, 78, 79, 80, 86, 88, 85, 87];
const R16_TREE_ORDER = [89, 90, 93, 94, 91, 92, 95, 96];
```
A `byTreeOrder(order)` comparator sorts each round's matches by index in these arrays before rendering. QF/SF/Final don't need an explicit order constant — their feeder relationships pin them implicitly. If the bracket ever looks like it's sending a winner to the wrong next match, check these arrays against `BRACKET_TEMPLATE`'s `[W##]`/`[L##]` placeholders first.

### Bracket Connector Lines
Visual lines connecting each round's slots to the next round are pure CSS (`styles.css`, "BRACKET CONNECTOR LINES" section) — `::before`/`::after` pseudo-elements on `.b-slot` forming elbow connectors, using `--bconn`/`--bconn-c` custom properties and `:nth-child(odd)`/`:nth-child(even)` rules to route the corner radius correctly depending on whether a slot is the top or bottom feeder into the next round's (taller) slot. No JS involved — relies entirely on the slot-height-doubling layout described in "Bracket Slot Layout" above.

### Bracket Card Venue Display
Bracket cards (`.b-match`, via `bMatchHtml()`) show city only, not the full "Stadium, City" venue string — `venueCity(venue)` splits on the last comma and keeps the trailing segment. This is scoped to the bracket only; Schedule cards, the match detail modal, and notifications still show the full venue string since they have more room and benefit from the stadium name.

### Live-Mode "Speculative" Indicator
When `state.liveMode` is on, any card whose displayed content could differ from Official mode gets a green/blue gradient border (`.live-affected` class, same `--grad-green` gradient as the live-mode toggle button itself — see CSS rule in styles.css just after `.match-card.live`). Distinct from `--grad-live` (red/orange), which marks a match currently in-play, not a speculative/live-mode-derived value.
- **Standings (per-group cards)**: bordered only when that specific group has a `IN_PLAY`/`PAUSED` match right now (`groupHasLiveMatch(group)`) — Official vs Live can only disagree for a group while one of its matches is actually live.
- **Third-Place Wildcard Rankings card**: bordered whenever *any* group has a live/paused match (`anyGroupHasLiveMatch()`) — the wildcard ranking draws from all 12 groups' 3rd-place teams at once, so a live match anywhere can reshuffle it.
- **Round of 32 cards (both Bracket and Schedule views)**: broader rule than the above two, because R32 slots resolve directly off group standings (every later bracket round resolves off an actual match's W/L, so it can't disagree). Bordered whenever `state.liveMode` is on **unless both sides are already locked in** via `applyBracketResolutions()`'s `_homeBracketResolved`/`_awayBracketResolved` flags (`r32MatchLiveAffected(match)` in `app.js`). Those flags are computed off FINISHED-only matches (`computeGuaranteedPositions()`), so they're identical in both Live and Official mode — once a side is locked, it stays unbordered regardless of what's live elsewhere. Until locked, the border shows even with no match currently live in the feeding group, since an already-FINISHED result elsewhere can still shuffle group position differently under Live vs Official. This intentionally does **not** reuse `groupHasLiveMatch()` — that check only catches "a match is live right now," which is too narrow for R32 (see Known Issues if this regresses).

### Live Clock
`getMatchMinute()` prefers ESPN clock data when available:
- `elapsedSec = _espnClock + (Date.now() - _espnFetchedAt) / 1000`
- Period 1: `Math.floor(elapsedSec / 60)` → capped at `45+N′`
- Period 2: same → capped at `90+N′`

Falls back to `firstHalfStart`/`secondHalfStart` timestamps from `data.json` if ESPN clock data is absent.

`update_tracker.js` still sets `firstHalfStart`/`secondHalfStart` as before — these remain valid fallback anchors.

### Official vs Live Toggle
- **Official**: standings computed from `FINISHED` matches only
- **Live**: standings computed from `FINISHED` + `IN_PLAY` + `PAUSED` (uses current score)
- Toggle affects: Standings view, 3rd-place wildcard table, Bracket placeholder resolution, and (since `state.liveMode` is global) Round of 32 placeholder resolution on the Schedule view too — see "Live-Mode 'Speculative' Indicator" above for how each surface signals this visually.
- State stored in `state.liveMode` (boolean), toggled via the `#live-mode-fab` floating action button
- Fair play points need no special-casing for this toggle: `homeFairPlay`/`awayFairPlay` are summed inside the same `includeStatuses`-gated loop as pts/gd/gf in `computeStandings()`, so in-progress cards are picked up automatically in Live mode and ignored in Official mode, just like the current score.

### Fair Play Tiebreaker
Group standings (and the 3rd-place wildcard ranking) use FIFA's disciplinary-points tiebreaker as the 4th sort key, after points → goal difference → goals for and before alphabetical fallback: 1 yellow card = `-1`, an indirect red (second yellow) = `-3`, a straight red = `-4`, a yellow plus a straight red in the same match = `-5`. Less negative wins.

- **Source**: football-data.org has no card data at all, so this is computed entirely from ESPN's `competitions[0].details[]` — the same array already used for goal scorers and the live stats pill. `classifyMatchFairPlay(details, ourHomeTeamId)` groups card events by athlete within a match to tell straight reds apart from second-yellow dismissals. Implemented identically in both `app.js` and `scripts/update_tracker.js`.
- **Per-match, not per-team-total**: each match stores `homeFairPlay`/`awayFairPlay` (the deduction for that match only, e.g. `-1`, `-4`), exactly like `homeScore`/`awayScore`. `computeStandings()` sums these into each team's `fairPlayPoints`.
- **Frontend (live)**: `mergeESPNData()` in `app.js` derives `homeFairPlay`/`awayFairPlay` from the same 10s ESPN poll already used for live scores/stats, for any match it's actively tracking — a live preview that self-corrects on the next poll as `details[]` fills in (same lag behavior as goal events). `ESPN_AUTHORITATIVE_FIELDS` in `fetchData()` includes both fields so data.json never clobbers the live ESPN value while a match is tracked.
- **Backend (permanent)**: `update_tracker.js`'s `syncMatchStats()` fills in every permanent per-match stat field (cards, fouls, corners, saves, passPct, shots, shots on target, tackles, interceptions, clearances, crosses, long balls, and — Group Stage only — fair play points) for any `FINISHED` match where `isStatsComplete()` says a field is still missing. `resolveEventContext()` resolves the ESPN event two ways: if the match already has a persisted `espnEventId` (set by a prior sync, or by the frontend's live merge and carried through `data.json`), it's a **single** `/summary?event=` call — `header.competitions[0].details[]` gives the card events, `boxscore.teams[].statistics[]` gives every other stat, both from the one response. Otherwise it falls back to a discovery search across `?dates=` for the kickoff date and the two adjacent days (absorbing the case where a near-midnight-UTC kickoff lands in a different ESPN date bucket than our PT-converted date string), finds the event via `findESPNEvent()` (ESPN team ID first, `ESPN_NAME_MAP` fallback), and persists the resulting `espnEventId` onto the match so every later sync for that match takes the single-call path. Card events and boxscore stats never change after the final whistle, so once `isStatsComplete()` is true a match is never re-fetched. This required porting `espnId` (added to every `TEAM_MASTER_DATA` entry), `ESPN_NAME_MAP`, and ESPN-fetch helpers into `update_tracker.js`, which previously had no ESPN integration at all.
- **Retroactive backfill note**: `isStatsComplete()` checks every field the match is expected to have (not just `homeYellowCards`, which was the original — too-narrow — completeness signal). This matters in practice: when `homeShots` etc. were added, every previously-"complete" match was missing them, so they'd have stayed permanently unfetched under the old single-field check. Worse, a handful of older matches had **literal `0` baked into `homeSaves`/`awaySaves`** from before `parseBoxscoreStat()` was changed to return `undefined` on a miss — since `0` is a number, `typeof m.homeSaves === 'undefined'` never re-triggered for them, so they stayed wrong forever even after that fix shipped. Confirmed against live data.json: 24 of 26 already-"synced" finished matches had `homeSaves: 0, awaySaves: 0` baked in, while ESPN's own `/summary` endpoint for those same matches (e.g. event 760415, Mexico v South Africa) still has the real values (`saves: 2` each). Extending `isStatsComplete()` to gate on `homeShots` too causes all of these to be re-fetched and corrected as a side effect, since `syncMatchStats()` always overwrites every boxscore-derived field (including `homeSaves`/`awaySaves`) whenever a match is re-processed — no separate one-off repair script needed.
- **Known limitation**: ESPN's `details[]` doesn't flag "this red card was a second yellow" explicitly. When a single athlete has exactly one yellow and one red logged in the same match, `classifyMatchFairPlay()` assumes it's an indirect red (`-3`) rather than yellow + straight red (`-5`), since that's the far more common real case — this hasn't been validated against a confirmed second-yellow example yet.

### FIFA World Ranking & Head-to-Head Tiebreaker
Full official FIFA group-stage tiebreaker order, as implemented: points → goal difference → goals scored → head-to-head mini-league among tied teams → fair play points → FIFA World Ranking position → alphabetical (last resort — FIFA's actual final tiebreaker is a literal drawing of lots, which can't be replicated programmatically).

- **`fifaRank`**: a static field added to every entry in `TEAM_MASTER_DATA` (both `app.js` and `update_tracker.js`), e.g. `"Argentina": { group: "J", iso: "ar", espnId: 202, fifaRank: 1 }`. Lower number = higher-ranked team. This is now only a **fallback default** — see "Refreshing" below for the live source. `fifaRankOf(team)` returns `9999` for any team missing from both the live data and `TEAM_MASTER_DATA` (defensive fallback, should never trigger since all 48 teams are populated).
- **Data source**: manually-saved HTML snapshot of FIFA's official ranking page (`blueprint_data/fifa_rankings_YYYY-MM-DD.html`, currently `2026-06-17`), since `inside.fifa.com`'s backend API is undocumented and the legacy `fifa.com/api/ranking-overview` endpoint is dead (returns a custom 404 page, not a bot block). Parsed via regex against stable CSS class patterns (`custom-rank-cell_rankNumber__*`, `fifa-world-ranking/{CODE}?gender=men`, `custom-points-cell_points__*`). Matched to our team names via FIFA's 3-letter codes (not display names — 7 of 48 teams have name mismatches, e.g. FIFA's "Korea Republic" vs. our "South Korea", "IR Iran" vs. "Iran", "Cabo Verde" vs. "Cape Verde Islands").
- **Refreshing (automated)**: `src/data/fifa_rankings.json` is a generated data file (same pattern as `combinations.json` — fetched once at startup by `fetchFifaRankings()` in `app.js`, stored in `state.fifaRankings`) holding `{ asOf, source, ranks: { team: rank } }`. `fifaRankOf(team)` in both `app.js` and `update_tracker.js` checks this live data first, falling back to `TEAM_MASTER_DATA[team].fifaRank` only if the team is missing from it. `update_tracker.js`'s `syncFifaRankings()` runs at the top of every cron tick (`main()`): it lists `blueprint_data/`, filters to `fifa_rankings_YYYY-MM-DD.html` files, takes the lexicographically-last filename (dates sort correctly as strings), and compares it against the `source` field already baked into `fifa_rankings.json` — if there's a newer one, it re-parses it via `parseFifaRankingsHtml()` and overwrites `fifa_rankings.json` (committed alongside `data.json` in the same git push, only when something actually changed). **To refresh mid-tournament, just drop a new dated HTML snapshot into `blueprint_data/` on `main`** — no code changes or manual `TEAM_MASTER_DATA` edits needed; the next Actions run (within 5 minutes) does the rest.
- **Head-to-head mini-league** (group standings only): `sortStandingsWithHeadToHead(teams, groupMatches)` first sorts by pts → gd → gf, then walks the sorted list clustering consecutive teams with identical pts/gd/gf. Clusters of size 1 pass through unchanged. For clusters of 2+, `resolveHeadToHead(cluster, groupMatches)` builds a mini standings table using only matches played between cluster members, sorts by that mini-league's pts → gd → gf, and falls through to fair play → FIFA ranking → alphabetical if the mini-league still can't separate them (handles FIFA's documented edge case: a closed cyclic tie, e.g. three teams each winning one and losing one against each other, where head-to-head mathematically cannot discriminate).
- **Cross-group rankings** (3rd-place wildcard) skip head-to-head entirely since the tied teams are in different groups and never played each other — straight to fair play → FIFA ranking → alphabetical.
- Implemented identically in `app.js` and `update_tracker.js` (`fifaRankOf`, `sortStandingsWithHeadToHead`, `resolveHeadToHead`), called from `computeStandings()` in both files.

## API Integration (football-data.org)

It is **not** consulted for status or score on any recurring sync — see "Score Source of Truth" below.

## Score Source of Truth
Live `status`/`homeScore`/`awayScore` for every match — group stage and knockout alike — is driven entirely by ESPN on the backend (`syncLiveScores()` in `update_tracker.js`), the same source the frontend already trusts live. football-data.org used to fill this role (the main per-tick sync loop copied `apiMatch.status`/`score` straight onto each match), but its `score.fullTime` for a penalty-decided knockout match was confirmed corrupted: it reports the AET score *plus* each side's shootout goals scored, not the tied AET score (e.g. AET 1-1, shootout 3-4 on penalties, persisted as `"4-5"`) — which broke the tied-score signal `isStatsComplete()` needs to detect a shootout at all, for exactly the matches that needed it most. Rather than patch around that corruption again, football-data.org was removed from the live-status/score path entirely.

- `syncLiveScores(matches, dateCache, now)` skips any match already `FINISHED` (permanent from there) or still showing a bracket placeholder team name (`[1A]`, `[W74]`, etc. — nothing to look up yet). For every other match it resolves the ESPN event (single `/summary?event=` call via a cached `espnEventId`, or a `?dates=` discovery search the first time, identical pattern to `resolveEventContext()`/`syncMatchStats()`), maps ESPN's status via `ESPN_STATUS_TO_OURS` (kept in sync with `ESPN_STATUS_MAP` in `app.js`, including the `STATUS_DELAYED` → `PAUSED` fix from Known Issues #13 and the `STATUS_END_OF_EXTRATIME`/`STATUS_SHOOTOUT` → `IN_PLAY` mapping from "Penalty Shootouts"), and sets `homeScore`/`awayScore` straight from ESPN's competitor `score` field — which, unlike football-data.org's, is correctly tied for an in-progress or finished shootout.
- `LIVE_SYNC_WINDOW_MS` (36h) skips matches whose kickoff is still further out than that: football-data.org's old bulk endpoint covered every match for one HTTP call regardless of status, but this is a per-match ESPN call, so polling a fixture days before it can possibly start would be pure waste.
- `dateCache` (an ESPN `?dates=` → events Map) is created once per `main()` run and shared between `syncLiveScores()` and `syncMatchStats()`, so a match that flips to `FINISHED` mid-tick already has its `espnEventId` cached for the stats-backfill pass that follows in the same run — no duplicate discovery search.
- `resolveBracketPlaceholders()`'s shootout tiebreaker (see "Penalty Shootouts" below) and the frontend's `matchOutcome()` both already expected a tied score + separate shootout score — neither needed any change; only the upstream score source did.

### Kickoff & Venue Self-Heal
`kickoff`/`venue` used to be write-once: group-stage matches get them from football-data.org's bootstrap response, knockout matches from `BRACKET_TEMPLATE`'s hardcoded guess (written before FIFA/ESPN finalize the actual time/venue for that bracket slot), and nothing ever revisited either field afterward — so a later FIFA schedule correction (e.g. moving a kickoff by an hour, confirmed for the Ivory Coast v Norway R32 match, 2026-06-30) would never reach `data.json`. `syncLiveScores()` now self-heals both fields on every tick it already runs for a match (same loop, no extra fetch in the common case):
- **Kickoff**: `comp.date` (ESPN's own scheduled time for the competition) is compared against `m.kickoff` by timestamp (not string equality, since formats differ) and overwrites it on any mismatch.
- **Venue**: sourced from `found.comp.venue` in the discovery-search branch (the scoreboard endpoint's competition object carries it directly), or `summary.gameInfo.venue` in the cached-`espnEventId` branch — venue is **not** present on `summary.header.competitions[0]`, unlike `date`, which is. Formatted via `formatEspnVenue()` to match our `"{venue name}, {city}"` convention (ESPN's `address.city` includes the state for US venues, e.g. `"East Rutherford, New Jersey"` — only the part before the first comma is kept).
- Runs for every match `syncLiveScores()` already visits (group stage and knockout alike, gated by the same `FINISHED`/placeholder-team/`LIVE_SYNC_WINDOW_MS` checks as status/score), so this requires no separate pass and self-corrects within one ~5-minute sync cycle of FIFA/ESPN updating their own schedule.

## PWA / Service Worker

The app is installable (manifest.json + icons) and registers `src/sw.js` (`navigator.serviceWorker.register('/world_cup_dashboard/sw.js')` near the bottom of `app.js`).

### Caching strategy
- **Data/API requests** (`data.json`, ESPN hostnames, football-data.org) — network-first, falling back to cache only if the fetch fails.
- **Everything else** (`index.html`, `app.js`, `styles.css`, `vendor/idiomorph.esm.js`, icons, manifest) — **cache-first, with no network revalidation**. Once an asset is cached under the current `CACHE` version, it is served from cache indefinitely until that version changes.

### CRITICAL: bump the cache version on every static-file change
`CACHE` in `src/sw.js` is a hardcoded version string (`wc2026-vN`). The `activate` handler deletes old-versioned caches and `install` re-precaches everything — but only when a **new version of `sw.js` itself** is detected by the browser. If `sw.js`'s bytes are unchanged, no install/activate cycle runs at all, even if `styles.css`/`app.js`/`index.html` changed and were deployed.

**Whenever you change `src/styles.css`, `src/app.js`, `src/index.html`, `src/vendor/idiomorph.esm.js`, `src/manifest.json`, or any file in `src/icons/`, bump the `CACHE` version in `src/sw.js` (e.g. `wc2026-v6` → `wc2026-v7`) in the same change.** For small bug fixes, a minor decimal bump is fine (e.g. `wc2026-v11` → `wc2026-v11.1`) — the version just needs to change, it doesn't need to be a new integer. Forgetting this means every client that has previously visited the site — desktop browser, mobile browser, and the installed Android PWA alike — keeps serving the old cached assets forever, with no visible error; the change will look like it "didn't do anything," even though it deployed correctly. (This happened: the mobile header layout, schedule-data-leak, and commentary-race fixes all shipped without a version bump, so they silently failed to reach already-visited clients until `CACHE` was bumped to `v6`.)

`skipWaiting()` + `clients.claim()` are already wired up in `install`/`activate`, so once the version *is* bumped, the new worker takes over without requiring users to fully close/reopen the app.

### CRITICAL: update the version footer on every change

`src/app.js` has two constants near the top:
```js
const APP_VERSION = 'v8';
const APP_UPDATED = '2026-06-17 16:17 UTC';
```
Rendered into `<footer id="app-footer">` (in `src/index.html`, styled in `src/styles.css`) on init, as "WC2026 Dashboard vN · Updated <timestamp>". This exists so the user can confirm — by reloading the live site — that a deploy actually reached them, independent of whether they can see the underlying change. It's the same problem the service worker cache-bust rule solves, so treat it as one combined step:

**On every commit that changes any file under `src/`, bump `APP_VERSION` (matching the new `src/sw.js` `CACHE` version, e.g. both go `v8` → `v9`) and set `APP_UPDATED` to the current UTC date/time (`date -u +"%Y-%m-%d %H:%M UTC"`).** After merging to `main` and the Actions deploy completes, report the new version number to the user so they can verify the live site shows it. For a small bug fix rather than a feature change, prefer bumping the minor/decimal portion (e.g. `v11` → `v11.1`) instead of the next whole integer — reserve whole-number bumps for features/redesigns.

### Android system notifications
`sendSystemNotification()` calls `serviceWorkerRegistration.showNotification()` (real OS-level notification, works while the PWA/tab is alive in the background but not once Android fully kills the process) alongside the existing in-page toast from `queueNotif()`. Gated on `Notification.permission === 'granted'`. The bell icon (`#notif-permission-btn`, managed by `initNotifPermissionBtn()`/`updateNotifPermissionBtn()`) lets the user opt in; hidden entirely if the browser lacks `Notification` or `serviceWorker` support.

## Known Issues / Watch Points
1. **ESPN name mismatches**: If a match isn't getting ESPN updates, check `ESPN_NAME_MAP` in `app.js`. Add the mapping and push to fix.
2. **ESPN scoreboard is date-scoped**: Only returns today's matches. Full 104-match schedule always comes from `data.json`.
3. **Group source of truth**: always use `TEAM_MASTER_DATA[team].group`, not `m.group`, when computing standings — see "Group Source of Truth" above
4. **Knockout matchId population**: R32+ matches start with `matchId: null`. Self-healed via name matching once the API returns them.
5. **Half-time timestamp source**: `firstHalfStart`/`secondHalfStart` are set by the Actions runner clock, not the API — accurate to within one 30-second polling interval. ESPN clock is preferred when available.
6. **Group standings sort**: always use `Object.keys().sort()` when iterating groups — key order in JSON is not guaranteed
7. **ESPN events lead the score**: `details[]` populates `_espnEvents` ~30s before the score field updates. The goal notification uses `_seenGoalEvents` (event count key) to fire early; `_seenGoals` (score key) prevents double-firing.
8. **Stale service worker cache**: any deploy that changes `styles.css`/`app.js`/`index.html`/icons without also bumping `CACHE` in `src/sw.js` will look like it had no effect on any previously-visited client (browser tab or installed PWA) — see "PWA / Service Worker" above. If a shipped UI change "isn't showing up," check this first before assuming a deploy or rendering bug.
9. **Fair play yellow+red ambiguity**: `classifyMatchFairPlay()` can't tell a straight red apart from a second-yellow dismissal when ESPN logs exactly one yellow and one red for the same athlete in the same match — it assumes second-yellow (`-3`) since that's the common case. Unvalidated against a confirmed real second-yellow example; revisit if a fair-play total looks off by 2 points.
10. **(Fixed 2026-06-17) Group G/H team swap**: `TEAM_MASTER_DATA` had Spain & Cape Verde Islands assigned to Group G and Iran & New Zealand assigned to Group H — backwards from FIFA's official lettering (verified against fifa.com and blueprint CSV cross-checks for all other groups). Corrected to G: New Zealand, Iran, Belgium, Egypt / H: Uruguay, Saudi Arabia, Spain, Cape Verde Islands, in both `app.js` and `update_tracker.js`, plus the cosmetic `group` field on the 6 affected matches and the `standings.G`/`standings.H` rows already committed in `data.json`. This swap was the root cause of the (now-removed) "Cross-Group Matches" documentation — see "Group Source of Truth" above. If group lettering ever looks suspicious again, sanity-check `TEAM_MASTER_DATA` against the live FIFA standings page, not just internal consistency.
11. **(Fixed 2026-06-17, v11.1) Commentary auto-resume never actually resumed**: the resume-to-latest logic (15s timer + the v11 timestamp fallback) correctly computed "go back to the latest comment," but then pinned `_commentarySeq` to that comment's concrete `sequence` value instead of clearing it. Since `_espnCommentary` keeps growing as new commentary arrives, that pinned value drifted further from index 0 with every new comment, and — because nothing re-arms the resume timer once the view is considered "caught up" — there was no further mechanism to catch up. The fix: whenever the view lands back on the latest comment (manual `›`, the 15s timer, or the fallback), `_commentarySeq` is now reset to `null`/`undefined` rather than snapshotted, so `commentaryInnerHtml()`'s default behavior (`idx === -1 → idx = 0`) keeps dynamically tracking whatever is actually newest. This reproduced consistently regardless of click pattern (single click, multiple clicks, or manually returning to latest) and regardless of whether the tab was foregrounded — all of which makes sense once you see that the bug never depended on timing at all.
12. **(Revised — original diagnosis was wrong) "ESPN's boxscore stats go empty for old matches"**: an earlier version of this note claimed `boxscore.teams[].statistics[]` permanently empties out a few hours after a match finishes, based on 24 finished matches all showing `saves: 0` for both teams. That diagnosis doesn't hold up: fetching `/summary?event=` directly for one of those same matches (Mexico v South Africa, a week-plus old) returns a fully populated ~28-stat boxscore including real `saves` values. The actual bug was in `syncMatchStats()`'s completeness check, which only looked at `homeYellowCards` to decide a match was "already synced" — so once cards/fouls were baked in (which always worked, since they're also derivable from the always-present scoreboard endpoint), the match was never revisited even though `homeSaves`/`awaySaves` had been incorrectly baked in as literal `0`/`0` by an earlier version of `parseBoxscoreStat()` (before it was changed to return `undefined` on a miss). Fixed by `isStatsComplete()` checking every persisted field, not just one — see "Fair Play Tiebreaker" → "Retroactive backfill note" above for the full account and how this also resolved itself without a one-off repair script.
13. **(Fixed 2026-06-22) Weather-suspended live match reverted to "upcoming"**: `ESPN_STATUS_MAP` in `app.js` mapped `STATUS_DELAYED` to `SCHEDULED`, intended for pre-match reschedules. But ESPN also reports `STATUS_DELAYED` for an in-progress match suspended mid-game (e.g. France v Iraq paused at halftime for thunderstorms), with `type.state` still `'in'`. Because the literal status-name lookup in `ESPN_STATUS_MAP` always takes precedence over the `typeState` fallback in `mapESPNStatus()`, the match got bumped back to `SCHEDULED` instead of `PAUSED` — the card showed it as upcoming, and the match detail modal showed "No match data available yet." (`hasScore` is `false` for `SCHEDULED`, gating both the score and the live data fetch). Fixed by mapping `STATUS_DELAYED` to `PAUSED`, matching `STATUS_DELAY`; genuine pre-tournament postponements/cancellations still route through `STATUS_POSTPONED`/`STATUS_CANCELED`.
14. **(Fixed 2026-06-30) Backend silently advanced the loser of any penalty shootout**: `update_tracker.js`'s `resolveBracketPlaceholders()` resolved `[W##]`/`[L##]` bracket placeholders with `homeWon = feeder.homeScore > feeder.awayScore`. For a knockout match decided on penalties, `homeScore`/`awayScore` stay tied at the AET score forever — `homeWon` was always `false`, so every shootout-decided match would have advanced its **loser** into the next round's bracket slot and permanently mislabeled the winner in `data.json`. Caught while building shootout support (see "Penalty Shootouts" above), before any real match had gone to penalties since the bug was introduced, so no bracket data was ever actually corrupted by it. Fixed by falling back to `homeShootoutScore`/`awayShootoutScore` (baked in by `syncMatchStats()` earlier in the same `main()` run) whenever `homeScore === awayScore`; resolution is deferred (`return null`, slot stays a placeholder) if the shootout score isn't available yet rather than guessing. The equivalent frontend logic in `app.js`'s `resolveTeam()` had the same shape of bug and was fixed the same way.
15. **(Fixed 2026-06-30) Backend permanent shootout-score backfill used a stat that never populates**: the original implementation of fix #14's prerequisite — baking in `homeShootoutScore`/`awayShootoutScore` so the bracket-resolution fix above has something to read — sourced the score from `boxscore.teams[].statistics[]`'s `penaltyKickGoals` stat, on the assumption (stated in "Penalty Shootouts" above at the time) that it would finalize once the match was `FINISHED`, even though it's known not to update live. Validated directly against the first two real WC2026 shootouts (Germany v Paraguay and Netherlands v Morocco, both decided 2026-06-29) via the `debug-espn.yml` debug harness: `penaltyKickGoals` stayed `"0"`/`"0"` for both teams in both matches even though both were fully `FINISHED` (`STATUS_FINAL_PEN`, `completed: true`). Had this shipped as-is, `homeShootoutScore`/`awayShootoutScore` would never have been backfilled for any real shootout, which in turn means fix #14's bracket-resolution tiebreaker would never have had data to read — every shootout-decided knockout match's `[W##]`/`[L##]` placeholder would have stayed unresolved forever. Fixed by switching `syncMatchStats()` to `parseShootoutScore()`, which counts `didScore === true` entries in the `/summary` endpoint's `shootout[]` array — the same field and computation the frontend's live preview already used correctly — and by extending `isStatsComplete()` to require `homeShootoutScore` whenever `wentToShootout(m)` (knockout stage, tied score) so a shootout match isn't considered "synced" until that field is actually populated. Cross-checked the shot-by-shot `shootout[]` data against ESPN's own `winner: true/false` competitor flag for both matches as an independent sanity check: Germany 3–4 Paraguay (Paraguay `winner: true`) and Netherlands 2–3 Morocco (Morocco `winner: true`) — both agree.
16. **(2026-06-30) football-data.org relegated to schedule scaffolding only — ESPN is now the sole live status/score source**: fix #15 above traced a real shootout-score bug back to `score.fullTime`, but the deeper issue was upstream of any single field: football-data.org's per-tick sync loop in `main()` copied `apiMatch.status`/`score` straight onto every match, and for a penalty-decided knockout match that `score.fullTime` is the AET score *plus* each side's shootout goals (e.g. AET 1-1, shootout 3-4 on penalties, persisted as `"4-5"`) rather than the tied AET score football-data.org's own UI shows — corrupting exactly the tied-score signal `isStatsComplete()`/`resolveBracketPlaceholders()` need to detect a shootout in the first place. Having now patched around football-data.org's score reporting twice in one day (fix #14's bracket tiebreaker, fix #15's backfill source), and since the frontend already trusts ESPN exclusively for live `status`/`homeScore`/`awayScore` with no equivalent corruption ever observed, football-data.org was removed from the live-score path entirely rather than patched a third time. It's now schedule scaffolding only: `bootstrapFromApi()` (full group-stage schedule, kickoff times, venues, team names, `matchId`) and the ongoing `matchId: null` self-heal for knockout matches once the API starts returning them. A new `syncLiveScores(matches, dateCache, now)` — the backend's equivalent of the frontend's `fetchESPN()`/`mergeESPNData()` — now drives `status`/`homeScore`/`awayScore` for every match with resolved (non-placeholder) team names, via `ESPN_STATUS_TO_OURS` (kept in sync with `app.js`'s `ESPN_STATUS_MAP`, including the `STATUS_DELAYED`→`PAUSED` fix from #13 and the shootout-status mappings from "Penalty Shootouts"). Since this trades football-data.org's one-bulk-call-covers-everything model for a per-match ESPN call, `LIVE_SYNC_WINDOW_MS` (36h) skips any match whose kickoff is still further out than that — without it, every fixture still days away would get hit on every 30s tick for a status that's guaranteed to still be `SCHEDULED`. `main()` now creates one `dateCache` per run and passes it to both `syncLiveScores()` and `syncMatchStats()`, so a match that flips to `FINISHED` mid-tick already has its `espnEventId` cached for the stats-backfill pass that follows in the same run. See "Score Source of Truth" above for the full design.
17. **(Fixed 2026-06-30) Knockout kickoff/venue were write-once, with no self-heal**: user reported the dashboard showing the Ivory Coast v Norway R32 match (matchNum 78) at 11 AM PDT while Fox Sports/ESPN showed 10 AM PDT. Traced to `BRACKET_TEMPLATE`'s hardcoded `kickoff`/`venue` for that slot being written into `data.json` exactly once, at bootstrap (`kickoff: tmpl.kickoff`) — unlike `matchId`/status/score, nothing ever revisited either field afterward, so a later FIFA schedule correction had no path back into `data.json`. Fixed by extending `syncLiveScores()` to self-heal both fields from ESPN's own `comp.date` and venue data on every tick it already runs for a match — see "Kickoff & Venue Self-Heal" under "Score Source of Truth" above. This sandbox's network egress policy blocked direct verification of ESPN's live value for event 760490 at the time of investigation, so the fix addresses the architectural gap rather than confirming which specific time was correct.
18. **(Fixed 2026-07-01) Dashboard stalled/mislabeled across every regulation↔extra-time break**: user reported the live dashboard visibly stuck at `90'+8'` for several minutes before Belgium v Senegal's extra time started, then briefly mislabeled the break between the two extra-time halves as plain `HT`. Root cause: `getMatchMinute()`/`statusBadge()`/`matchCardHtml()` each independently hardcoded `PAUSED` → `"HT"` with no check on `_espnPeriod`, so every non-regulation-half-time break looked identical to (or, worse, invisible relative to) real half-time. Fixed by centralizing the label in `pausedStatusLabel(match)` and adding `_regulationEnded` (set from `data.keyEvents[]`'s `"end-regular-time"` marker, a field in the `/summary` endpoint we were already fetching for commentary but never read) as the signal that regulation just ended and an ET/pens decision is pending. See "Extra-Time Transition Gap" above for the full design. This sandbox's network egress policy blocked direct verification of ESPN's live status/period values during the actual Belgium v Senegal gap (same limitation as #17), so `scripts/snapshot_extra_time.js` + `.github/workflows/snapshot-extra-time.yml` were added to automatically capture raw scoreboard/summary payloads the next time any match approaches or enters extra time, rather than relying on someone noticing in time to run the existing `workflow_dispatch`-only `debug-espn.yml` harness.
19. **(Fixed 2026-07-02) Live clock hung on "120'" through ~12 minutes of extra-time stoppage**: same Belgium v Senegal match, second extra-time half — the dashboard froze on a bare `"120'"` instead of showing added time, and reloading the page didn't help. Confirmed directly from data already in `blueprint_data/` (no new capture needed): ESPN's raw `status.clock` field caps at the period boundary during stoppage and never advances past it — several already-`STATUS_FULL_TIME` matches show `clock: 5400` (90:00) sitting alongside a correctly-incrementing `displayClock: "90'+9'"`. Periods 1/2 already prefer `displayClock`'s `"+N"` suffix during stoppage so this never surfaced there, but the 2026-06-29 ET clock fix (`08a5083`) had deliberately turned that fallback *off* for periods 3/4 (ET's `displayClock` was observed to keep anchoring to `"90'+N'"` instead of relabeling to 105'/120'), leaving raw-clock-only — which inherits the same capping with nothing to catch it. Also considered building out `scripts/snapshot_extra_time.js` into a self-relaunching harness to empirically confirm ESPN's exact ET `displayClock` behavior, but concluded it wasn't load-bearing for the fix (see below) and would have added CI complexity for a "nice to confirm, not required" payoff — skipped. Fixed by tracking `_clockFrozenSince` in `mergeESPNData()` (the wall-clock moment the raw clock last actually changed) and having `getMatchMinute()` count real elapsed time from that point once the raw clock is stuck at/past the period cap, instead of freezing — an estimate rather than the referee's official added-time figure, but one that reflects reality regardless of what ESPN's ET `displayClock` does or doesn't report. See "Extra-Time Stoppage Clock Freeze" above.

## Open Issues / Pre-Restart Checklist

Written at 2026-07-19 shutdown, based on the user's own end-of-tournament experience plus a fresh read of the Known Issues log above. These are things to actually re-evaluate — not just re-read — before turning the automation back on for the next tournament. Nothing in this section has been fixed; it's a punch list, not a changelog.

1. **Extra-time/penalty clock handling was still fighting itself in practice, despite #18/#19 above being logged as "Fixed."** Both of those fixes were designed and shipped from captured JSON snapshots, not a confirmed live observation — the sandbox that built them had its network egress blocked during the actual Belgium v Senegal gap (noted inline in both entries). Live user experience across later knockout matches was that the in-app estimated clock (`_clockFrozenSince`) and ESPN's actual status/period transitions (`STATUS_END_OF_EXTRATIME` → `STATUS_SHOOTOUT`, the `_regulationEnded`/`pausedStatusLabel()` breaks) were still visibly out of sync at times. **Before relying on this again**: deliberately watch a real match go through regulation → ET → penalties end-to-end (or use `scripts/snapshot_extra_time.js`'s captures, which never actually got exercised against a full ET+pens sequence this tournament) and confirm the badge/clock behavior holds up, rather than trusting the existing fix.
2. **Notification replay has no time gate — never fixed, not previously documented.** `queueNotif()` fires purely off state diffs (`_seenGoalEvents`/`_seenGoals`/status transitions), with nothing checking how long ago the underlying event actually happened. `resyncNow()` (triggered on PWA resume — `visibilitychange`, `online`, `pageshow` with `persisted`) forces an immediate `fetchData()` + `fetchESPN()`; if you'd been away for a while — sometimes past full time — the diff against stale in-memory state fires a burst of goal/full-time modals+sounds all at once for things that already happened. **Needs**: a guard in the notification path that skips queuing anything whose underlying event is older than a few minutes (compare against the event's own timestamp where available, or `data.lastUpdated`/`Date.now()` as a fallback), added before this goes live again.
3. **Score/stats source of truth is currently well-defined — but re-verify it, don't assume it carries over.** "Score Source of Truth" above documents the end state clearly: ESPN alone drives live `status`/`homeScore`/`awayScore` on both frontend and backend (`syncLiveScores()`), football-data.org is schedule-scaffolding only (`bootstrapFromApi()` + `matchId` self-heal). That division of labor was earned through three real bugs (#14–#16) during WC2026 and should **not** need to be re-litigated — but it *should* be re-confirmed against whatever schedule-data API is used for the next tournament, since nothing guarantees football-data.org (or its scope/behavior) is even the right scaffolding source for a different competition.
4. **ESPN endpoint slug is hardcoded to the men's World Cup**: `ESPN_SCOREBOARD_URL` uses `soccer/fifa.world`. This was never validated against any other competition. If the next tournament is the Women's World Cup (or anything else), confirm the correct ESPN league slug before assuming `fetchESPN()`/`syncLiveScores()` will resolve any matches at all — a silently-empty scoreboard response would look like "ESPN just isn't returning anything" rather than an obviously wrong URL.
5. **Tournament-format mismatch is the big structural risk, not a data-swap.** `BRACKET_TEMPLATE`, `combinations.json`, `SLOT_TO_OPPONENT`, `computeThirdPlaceRankings()`, and the `R32_TREE_ORDER`/`R16_TREE_ORDER` constants are all built specifically for WC2026's 48-team / 12-group / Round-of-32-with-8-of-12-third-place-wildcards format. The current Women's World Cup format (32 teams, 8 groups, straight to Round of 16, no third-place wildcard mechanism at all) does not map onto this structure. **Before reusing this codebase for a differently-shaped tournament**, treat the bracket/wildcard logic as needing a rework, not a `TEAM_MASTER_DATA` refresh — confirm the actual field size and format of the target tournament first, since it changes what code exists at all, not just what data populates it.
6. **Fair-play yellow+red ambiguity (#9 above) was never validated** against a confirmed real second-yellow case across the entire tournament. Still an open guess (`classifyMatchFairPlay()` assumes second-yellow over straight-red+yellow when it sees exactly one of each for one athlete). Watch for it if a fair-play total looks off by 2 points next time.
7. **`TEAM_MASTER_DATA`, `fifaRank` defaults, and `blueprint_data/fifa_rankings_*.html` are now stale WC2026 data** (team list, groups, ESPN IDs, ranking snapshot) and will need a full refresh — not a patch — for a different tournament's actual participant list.
8. **Several fixes shipped without live verification** (#17 kickoff/venue self-heal, #18 extra-time transition labels, #19 ET clock freeze) because the dev sandbox's network egress was blocked during the actual live gap each was built to fix — each entry says so inline. Point 1 above already calls out the ET/penalty case specifically; the same "verify before trusting" caution applies to the kickoff/venue self-heal logic too, the first time a future tournament's schedule actually shifts mid-tournament.

### Iterating on a feature branch without touching the live site
`sync.yml`'s `deploy` job hardcodes `ref: main`, so there's no way to get a branch onto `gh-pages` without merging to `main` first. For UI/CSS/JS iteration, **`npx serve src/` against the checked-out branch is the fast loop** — ESPN polling is browser-direct and `data.json` is just a static file already committed in the branch, so everything works locally with no backend, no auth, and no deploy. Edit, refresh the tab, repeat. Only merge to `main` once a change is verified locally.

## Version Bumps
Per "PWA / Service Worker" above, `APP_VERSION`/`APP_UPDATED` (`src/app.js`) and `CACHE` (`src/sw.js`) need to move together on every `src/` change that reaches `main`. **While iterating on a feature branch, hold off on bumping these** until the user explicitly asks to merge/pull into `main` — bump them as the last step right before that merge, not on every intermediate commit, so the version number stays meaningful as "what's actually live."
